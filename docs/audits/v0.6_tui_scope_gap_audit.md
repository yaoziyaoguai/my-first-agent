## 1. 背景

v0.5.1 已完成 `runtime boundary evidence hardening` 并 tag 为 `v0.5.1`（指向 `240308b`，annotated tag object `ce65bdca`）。
进入 v0.6 规划时，只读盘点发现项目里 TUI **不是全新边界**，而是已有相当完整的资产：

- `agent/input_backends/textual.py`（549 行）— 已是 `display/input adapter`，**不是** dead code、**不是** prototype、**不是** mixed-concern；
- `agent/input_backends/simple.py`（129 行）— simple CLI fallback；
- `docs/V0_2_BASIC_TUI_PLAN.md`（208 行）— v0.2 已规划基础 TUI scope；
- `docs/V0_3_PLANNING.md`（269 行）— v0.3 规划高级 TUI（多面板 / timeline / event replay / Esc cancel / paste burst）；
- `tests/test_input_backends_simple.py` / `tests/test_input_backends_textual.py` — TUI 已有 tests 覆盖；
- `main.py` 已有 `INPUT_BACKEND_ENV` 切换、`--shell` 入口、双路径 `RuntimeEvent` renderer（simple CLI 与 Textual 共享同一条 RuntimeEvent 投影）。

因此本文档的目标**不是**写 "v0.6 TUI MVP plan"，而是先理清：
**"v0.6"这个标签对应到底应该做什么 / 不应该做什么 / 哪些 prior planning 仍然有效。**

## 4. Architecture boundary assessment

### 4.1 三层边界（**当前已基本健康**）

#### Runtime core
- 文件：`agent/core.py` / `agent/state.py` / `agent/checkpoint.py` / `agent/confirm_handlers.py` / `agent/response_handlers.py` / `agent/tool_executor.py`
- 负责：state transition / model output dispatch / pending confirmation dispatch / checkpoint/resume / tool execution orchestration
- 不负责：TUI 渲染 / 文本布局 / terminal widget / UI 状态
- 现状：✅ `agent/core.py` 不 import 任何 `input_backends.*`；与 TUI 完全解耦

#### Display / event adapter
- 文件：`agent/display_events.py`（470）/ `agent/runtime_events.py`（572）/ `agent/runtime_observer.py`（246）
- 负责：runtime event → view-friendly model；保持显示与 runtime 语义分离
- 不负责：runtime decision / checkpoint 写入 / handler 绕过 / tool execution
- 现状：✅ 已分层，schema 稳定（v0.5.1 已为 ModelOutputKind 4-value vocabulary 钉死真值源）

#### TUI layer
- 文件：`agent/input_backends/textual.py`（549）/ `agent/input_backends/simple.py`（129）/ `main.py` 的 `_run_textual_runtime_turn` / `_run_simple_cli_runtime_turn`
- 负责：渲染 plan / current step / runtime event / pending confirmation / tool summary；收集用户输入；返回 `UserInputEvent`
- 不负责：直接改 runtime state / 直接改 checkpoint / 绕过 confirmation handlers / 直接读真实 sessions/runs
- 现状：✅ textual.py 模块 docstring 已显式声明全部 NOT；`main.py` 已用 `INPUT_BACKEND_ENV` + `--shell` 切换并双路径共享 RuntimeEvent renderer

### 4.2 边界已健康的证据
1. `agent/core.py` 全文搜索 `input_backend` / `textual` / `TextualInput` → 0 命中（runtime → TUI 单向；TUI → runtime 反向依赖也只走公共 event/envelope）；
2. `textual.py` 顶部 import 只来自 `agent/display_events` 与 `agent/user_input`，不触碰 state/checkpoint/handler；
3. `main.py` 中所有 TUI 包装函数都标注 deprecation 桥接注释，主路径走 RuntimeEvent renderer。

> **结论**：v0.6 不需要"重新设计"三层边界 — 边界已存在。
> v0.6 的工作是**在既有边界上扩展行为，并补 characterization tests 防止扩展过程把边界悄悄打破**。

## 6. Recommended v0.6 strategy

**默认推荐**：v0.6 不直接叫 "TUI MVP implementation"。

更合理的切片顺序：

| Slice | 目标 | 类型 | 允许改 | 禁止改 |
|---|---|---|---|---|
| v0.6.0 | TUI scope alignment / gap audit（即本文档） | docs-only | 仅 `docs/audits/v0.6_tui_scope_gap_audit.md` | core / textual.py / tests / 既有 V0_2/V0_3 doc |
| v0.6.1 | characterization tests for existing `textual.py` & `simple.py` 行为 | tests-only | `tests/test_input_backends_textual.py` 扩 / `tests/test_input_backends_simple.py` 扩 | textual.py 本体 / core / handler |
| v0.6.2 | display adapter boundary assertion test | tests-only | 新建 `tests/test_display_adapter_boundary.py`（断言 core.py 不 import input_backends 等） | core / textual / display_events |
| v0.6.3 | 解 1 个最小 strict xfail（推荐先做 paste burst，因为不依赖 cancel_token） | production + tests | `agent/input_backends/simple.py` + 对应 test | core 主路径 / checkpoint / handler |
| v0.6.4 | Ask User / confirmation flow TUI characterization | tests-only | tests | production |
| v0.6.5 | release notes + audit doc + tag | docs-only | RELEASE_NOTES_v0.6.md + audit doc | 其余 |

**严禁打包**：把上述任意两个 slice 合并到一个 commit。

## 7. Next slice recommendation

**推荐**：**v0.6.1 — characterization tests for existing textual.py & simple.py 行为**。

理由：
- v0.6.0（本文档）一旦 commit + push，下一步必须先有"防回归网"才能动 production；
- textual.py 549 行 / simple.py 129 行的行为在已有 tests 中是否完全覆盖未知 — 应先 characterize 再扩展；
- 这一步是 tests-only，**零 production 风险**；
- 与 v0.5.1 的"先 characterize 再 extract"节奏完全一致。

**不推荐**：
- ❌ 直接动 textual.py 加新 widget；
- ❌ 直接试图解 strict xfail（应先在 v0.6.1 把基线钉牢）；
- ❌ 大规模重构 main.py 的 deprecation 桥接（风险过高）；
- ❌ 在 v0.6 任何 slice 中读真实 sessions/runs。
