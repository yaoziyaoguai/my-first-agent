# v0.5.1 Checkpoint/Resume Runtime Boundary Audit

> 文档定位：这是一份面向 reviewer 的 **evidence 索引**。
> 它本身不引入任何功能、不修改任何 production code、不改任何 test。
> 阅读顺序建议：第 1 → 2 → 3 节（理解全貌） → 第 4 / 5 / 6 节（按边界查 evidence） → 第 9 节（reviewer checklist）。

---

## 1. 背景与目标

### 为什么有 v0.5.1
v0.5.0（tag `v0.5.0` → `32d4ca1`）已经把 **runtime / display / event boundary** 做了第一轮收口：
- runtime event 替代终端直接 print；
- observer log_event 签名钉死；
- 终端 user-facing 文案有 baseline。

但在 v0.5.0 落地时，`agent/core.py` 的 `chat()` 与 `_run_main_loop()` 内部仍残留两段
**"长 if-elif 链 + 状态判断 + 调度副作用"** 混在一起的代码。这两段代码：
- 没有独立 helper，回归只能靠端到端测试触发；
- 没有 characterization tests 钉死分支语义；
- 一旦未来要做 TUI / cancel / resume 重写，没有"行为基线"可对比。

v0.5.1 的目标因此被显式收窄为：
> **不加新功能。**
> **不重写 checkpoint/resume。**
> **不动 v0.5.0 的 runtime/display/event boundary。**
> **只做：把 chat loop 里两个关键 dispatch 边界提取成 helper，并补齐 checkpoint→resume→dispatch 端到端 evidence。**

这是一种典型的 **"先 characterize、再 extract、再 bridge"** 的小切片瘦身节奏，
目的是为后续 v0.6 TUI MVP 留下可信基线，而不是在 v0.5.1 阶段做大爆炸式重构。

### v0.5.1 不承诺什么
见第 8 节。最重要的一条：**v0.5.1 不重写 checkpoint/resume 的 production 行为**，
只是把"现在的行为"用 tests 钉死 + 把"现在已经写在 chat loop 里的 dispatch 形状"提成 helper。

## 6. Checkpoint/resume evidence

### 当前 checkpoint 保存 / 恢复的关键字段
- `meta`：source / status / current_step_index / pending_user_input_kind / pending_tool_name 等元信息（写在 `save_checkpoint` 的 `checkpoint_saved` 事件 payload 中）；
- `task`：`TaskState` 的 declared fields（含 `pending_*` 各类标志）；
- `memory`：`MemoryState` 的 declared fields；
- `conversation.messages`：消息序列。

`load_checkpoint_to_state` 在恢复时使用 `_filter_to_declared_fields`：
- 未在 dataclass 上声明的字段会被**静默丢弃**；
- 这是 schema 演化的安全网，但也意味着字段重命名 / 删除时必须先升 round-trip 测试。

### 为什么"字段保留测试"不等于"runtime dispatch 命中测试"
即使 JSON 里的 `pending_step_confirmation = True` 在 round-trip 后字段值仍然是 True，
也无法证明 **"下一次 `chat()` 真的会走进 step confirmation 分支并命中 `handle_step_confirmation`"**：
- chat() 顶层的 if-elif 顺序可能漂移；
- `_dispatch_pending_confirmation` 内部分支顺序可能漂移；
- `core.state` 模块全局变量可能没有被正确装回；
- 任何一处静默 bypass 都会让用户的待确认动作"消失"，而 round-trip 测试完全察觉不到。

### 为什么 73bb6b2 是 v0.5.1 的唯一缺口
在 73bb6b2 之前：
- in-process dispatch evidence ✅（cdd1427）
- 字段保留 evidence ✅（既有 8 + 14 测试）
- **save → load → install → chat() → 命中 handler** 的端到端 evidence ❌

73bb6b2 用 5 个测试把这条桥补齐，且只补这一条桥，刻意不扩 production。

### 为什么不读真实 sessions/runs
- 真实 `sessions/` 含用户对话语义内容，是隐私边界；
- 真实 `agent_log.jsonl` 含运行期日志，体积大、不稳定；
- 真实 `runs/` 含执行产物，与测试隔离原则冲突。

v0.5.1 测试一律使用：
- `tmp_checkpoint_path` fixture（monkeypatch `checkpoint.CHECKPOINT_PATH` 到 `tmp_path / "checkpoint.json"`）；
- `dispatch_probe` fixture（monkeypatch 5 个 confirm handler 为记录型 probe）；
- `SimpleNamespace()` 作为 fake `client`（`LoopContext` 拒绝 `None`，必须用显式 fake）；
- `_FakeResponse(stop_reason)` 作为最小 SDK response stand-in。

这些是 deliberate 的**测试边界约定**，不允许在 73bb6b2 之后被悄悄放宽。
